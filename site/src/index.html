A proxy for your LLM gateway

Ormuz sits between your code and the LLM gateway. It paces requests with a token bucket, queues bursts in FIFO, and retries upstream 429s so your tools stop hitting the rate limit.

What Ormuz does

Four jobs, executed by a single Node process you can run as a sidecar or as a system-wide proxy.

Token-bucket pacing

Per-key rate control with a configurable safety factor. Buckets start full, so short bursts pass; sustained traffic gets paced to your real RPM budget.

Bounded FIFO queueing

Bursts wait in a per-key queue with a hard depth cap and a projected-wait cap. When the line is too long, Ormuz fails fast with a local 429 instead of stranding the request.

HTTPS via CONNECT

Acts as a system-wide proxy. Tunnels TLS byte-for-byte without termination — no custom CA, no MITM. Per-host rate limits apply to every CONNECT.

Prometheus metrics

Queue depth, dequeue wait, tokens available, and outcome counters per bucket key. One scrape endpoint, ready for your existing dashboards.

How it works

One Node process, one scheduler, one path that's easy to reason about under load.

   client                    Ormuz                       LLM gateway
   ------     /v1/...     -----------------     pace      -------------
   tools  ─────────────▶  route → bucket  ─────────────▶  upstream API
   sdks                   ↓                                ▲
                          token bucket + FIFO queue        │
                          ↓                                │
                          429 retry on first attempt ──────┘

Each request is mapped to a bucket key, admitted by the scheduler, and forwarded with undici. If the gateway answers 429, Ormuz pauses that bucket, re-queues the request once, then gives up cleanly.

Drop it into the loop you already have

Terminal

curl -X POST "http://localhost:8787/v1/openai/chat/completions" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer REQUEST_PROVIDED_KEY" \
  -d '{
    "model": "gpt-4o-mini",
    "messages": [{"role":"user","content":"hello"}]
  }'

Same request shape as the upstream. Ormuz strips routing headers, forwards the rest, and streams the response back.

Install in 30 seconds

macOS today. Linux supported as a sidecar; system-wide install lands soon.

git clone https://github.com/grebmann1/ormuz-proxy.git
cd ormuz-proxy
npm install
npm run dev
# Ormuz now listens on http://localhost:8787

cd ormuz-proxy

Aside

Why "Ormuz"?

The Strait of Hormuz is the narrow chokepoint that about a fifth of the world's seaborne oil passes through. Tankers don't fight the strait — they line up, take their slot, and move on.

This proxy is the strait between your code and the LLM gateway: narrow on purpose, with a queue when traffic gets dense, and the ships still get through.
